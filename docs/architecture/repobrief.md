# RepoBrief

RepoBrief is the public system name for this repository snapshot and agent-briefing tool.

The historical repository and Python package name remains `lenskit` for now. That name is a compatibility surface, not the preferred product name. Phase 1 is a brand and vocabulary migration only: no package rename, no schema kind rename, no import-path migration, and no invalidation of existing dumps.

## Purpose

RepoBrief turns a repository state into a deterministic, citable brief for humans, coding agents, reviews, and later MCP clients.

A RepoBrief run may produce a Brief Bundle containing:

- a Canonical Brief Source,
- a Brief Bundle Manifest,
- an Agent Brief / Reading Pack,
- citation and range surfaces,
- retrieval indexes,
- card surfaces,
- health and surface diagnostics,
- optional profile-specific reports such as export safety.

## Core model

The Canonical Brief Source is the sole content authority inside a generated brief.

Brief Sidecars are navigation, diagnostic, evidence-index, or cache surfaces. They help a consumer find and inspect the Canonical Brief Source. They do not replace it.

A Brief Bundle is a snapshot at generation time. It does not automatically represent the current working tree, the latest GitHub state, or a pull request diff unless that provenance is explicitly present and checked.

## Allowed capabilities

RepoBrief may:

- explicitly create repository snapshots,
- read existing Brief Bundles,
- locate artifacts by role,
- resolve required reading for task profiles,
- report health, freshness, and availability,
- resolve ranges and citation surfaces,
- query existing indexes,
- later expose read-first MCP resources and a small set of explicit tools.

## Non-goals

RepoBrief must not:

- trigger implicit refresh during read operations,
- mutate Git state as a side effect of reading,
- create pull requests,
- write patches,
- generate review verdicts,
- treat tests as release approval,
- claim runtime correctness,
- claim test sufficiency,
- introduce LLM inference into the deterministic core,
- introduce embeddings or semantic reranking into the core.

## Negative semantics

New RepoBrief-facing artifacts must preserve explicit negative semantics. A successful artifact check does not establish:

- `truth`
- `correctness`
- `completeness`
- `runtime_behavior`
- `test_sufficiency`
- `regression_absence`
- `repo_understood`
- `claims_true`
- `forensic_ready`

## Read/create separation

RepoBrief has two distinct operation classes:

1. create operations, such as `snapshot create`, which may write new Brief Bundle artifacts;
2. read operations, such as `snapshot status`, `artifact get`, `required-reading resolve`, `range get`, and `query`, which must not write or refresh.

This separation is a core safety boundary. A stale or degraded brief must be reported as stale or degraded. It must not be silently regenerated by a read path.

## Naming scope

The public system name is RepoBrief.

The legacy implementation namespace remains `lenskit` until a later package and repository rename decision explicitly changes it.

Existing artifact kinds such as `lenskit.*` remain compatible during this phase. Renaming artifact kinds is outside the scope of this document.
